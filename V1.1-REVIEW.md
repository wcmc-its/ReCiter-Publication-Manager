## 2. JWT / Auth Changes (Phase 8)

### What changed

**`findUserPermissions()`** now returns a composite JSON object instead of a roles-only
array:

```json
{
  "roles": ["Curator_Scoped"],
  "scopeData": { "personTypes": ["academic-faculty"], "orgUnits": ["Anesthesiology"] },
  "proxyPersonIds": ["cwid1", "cwid2"]
}
```

Users without scope/proxy assignments get `scopeData: null` and `proxyPersonIds: []` —
same as before functionally.

**JWT token** now carries three separate claims:
- `token.userRoles` — roles array (JSON string), same as before, zero breakage to existing
  consumers (SideNavbar, index page, components all read this unchanged)
- `token.scopeData` — scope boundaries for Curator_Scoped users
- `token.proxyPersonIds` — list of personIdentifiers this user can curate on behalf of

**Middleware** (`src/middleware.ts`) was rewritten from brittle role-count comparisons
(`userRoles.length === 1 && userRoles.includes(...)`) to a clean capability-based model:

```typescript
const caps = getCapabilities(userRoles);
// caps.canCurate.all, caps.canManageUsers, caps.isCuratorSelf, etc.
```

**One breaking infrastructure dependency:** the new middleware uses
`getToken({ req, secret: process.env.NEXTAUTH_SECRET })` instead of the old
`jose.decodeJwt()` (which required no secret). **If `NEXTAUTH_SECRET` is not set as a
runtime environment variable in the K8s deployment, every authenticated user will be
redirected to `/login` in a loop.**

See open question #1 below.

### What did not change

- The SAML cookie-bridge flow is untouched
- `authOptions`, `CredentialsProvider`, and the SAML `authorize()` handler are preserved
- All existing session consumers read `session.data.userRoles` as before

## 3. New Roles: Curator_Scoped and Proxy

### Curator_Scoped

A new role that restricts a curator to a subset of people based on:
- **Person types** (e.g., `academic-faculty`, `academic-counselor`) — OR logic within types
- **Organizational units** (e.g., `Anesthesiology`) — AND logic across dimensions

These boundaries are stored as JSON columns on `admin_users`:
- `scope_person_types` — nullable JSON array
- `scope_org_units` — nullable JSON array
- `proxy_person_ids` — nullable JSON array

The DDL migration (`scripts/migrations/add-scope-proxy-columns.sql`) has been applied to
the dev database. Prod migration is pending (see open question #2).

### Proxy delegation

A Curator_Scoped or Curator_All user can delegate curation rights to other registered
users. On the curate page (`/curate/:id`), authorized users see a **Grant Proxy** button
that opens a modal to search and assign proxy users for that person.

The proxy system is designed so that:
- Proxy access is per-faculty-member, not global
- Proxy users get access on their next login
- Superusers and Curator_All can grant proxy via Manage Users

### Manage Users changes

The user list now shows:
- Role labels (e.g., "Curator_Scoped") in a Role column
- Inline scope summary for Curator_Scoped users
- Proxy count badge

### Scope filter on search page

Curator_Scoped users see a "Show only people in my scope" checkbox (checked by default).
Unchecking it shows all people in the system.

## 4. Open Questions

### Q1. NEXTAUTH_SECRET in K8s — Mahender (blocker before Phase 10 deploy)

The middleware now depends on `NEXTAUTH_SECRET` as a runtime environment variable. The
`kubernetes/k8-secrets.yaml` template does not include it. The `k8-buildspec.yml` injects
`NEXT_PUBLIC_RECITER_TOKEN_SECRET` as a Docker build ARG, but that is a different variable
used for a different purpose.

**To verify:** Run against the live cluster —
```bash
kubectl get secret reciter-pm-secrets -n reciter -o jsonpath='{.data}' | python3 -c "import sys,json; d=json.load(sys.stdin); print(list(d.keys()))"
```

If `NEXTAUTH_SECRET` is missing: add it to the K8s secrets (same value as
`NEXTAUTH_URL_INTERNAL` secret or a new random 32-char string). This must be in place
before deploying Phase 10.

**Does the current reciter-dev deployment have NEXTAUTH_SECRET? If so, how is it
injected?**
