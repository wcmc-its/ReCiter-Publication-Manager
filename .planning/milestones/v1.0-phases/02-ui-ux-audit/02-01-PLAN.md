  <action>
1. Install eslint-plugin-jsx-a11y as direct devDependency:
   ```
   npm install --save-dev eslint-plugin-jsx-a11y@6.10.2
   ```

2. Update `.eslintrc.json` to this exact content:
   ```json
   {
     "extends": [
       "next/core-web-vitals",
       "plugin:jsx-a11y/strict"
     ],
     "settings": {
       "jsx-a11y": {
         "components": {
           "Button": "button",
           "Image": "img",
           "Link": "a"
         }
       }
     }
   }
   ```
   CRITICAL: Do NOT add `"jsx-a11y"` to a `plugins` array. The eslint-config-next already loads the jsx-a11y plugin. Only extend with `plugin:jsx-a11y/strict` in the extends array. Adding it to plugins would cause a "plugin already loaded" error.

   The `settings.jsx-a11y.components` mapping tells the linter that react-bootstrap's `Button`, Next.js `Image`, and Next.js `Link` components map to their semantic HTML equivalents, preventing false positives.

3. Run initial lint and capture baseline to a file:
   ```
   npx next lint 2>&1 | tee .planning/phases/02-ui-ux-audit/lint-a11y-baseline.txt
   ```

4. Count violations by rule from the baseline output. Document the total count and per-rule breakdown at the top of the baseline file as a summary comment block.

5. Do NOT fix any violations yet. Do NOT add eslint-disable comments. The baseline is diagnostic only.
  </action>

**Audit methodology per view (applies to all 4):**

A. **Code Review:** Read the component files listed above. Identify:
   - Accessibility issues: missing alt text, onClick on non-interactive elements, missing form labels, missing ARIA attributes, color contrast concerns
   - UX friction: confusing interactions, missing loading states, error states, empty states
   - Visual inconsistencies: spacing deviations from 02-UI-SPEC.md spacing scale (4/8/16/24/32/48px), typography deviations (allowed weights: 400, 600 only), color usage outside the defined palette
   - Performance concerns: large component rendering, unnecessary re-renders, missing memoization

B. **Live App Review:** Navigate to each view in the browser:
   - Take Chrome DevTools MCP screenshot and embed in report
   - Test keyboard navigation: tab through all interactive elements, verify logical tab order, check for visible focus indicators, check for keyboard traps
   - Test loading states: observe what appears while data loads
   - Test error states: if possible, trigger error conditions
   - Test empty states: if possible, navigate with no data

C. **Lighthouse Audit:** Run Lighthouse via Chrome DevTools MCP on each view:
   - Capture Accessibility score (target: 90+)
   - Capture Performance score (baseline, no target)
   - Record First Contentful Paint and Largest Contentful Paint
   - Document specific Lighthouse findings/opportunities

D. **ESLint Violations:** Filter the baseline lint output (lint-a11y-baseline.txt) for files related to each view. Document per-rule violation counts in the audit file.

E. **React Profiler:** Open React DevTools Profiler, trigger a render cycle on each view (e.g., navigate to it), capture:
   - Render duration
   - Number of components that rendered
   - Any notably slow components (>16ms render)
