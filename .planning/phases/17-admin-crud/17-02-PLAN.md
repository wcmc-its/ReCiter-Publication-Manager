Wait -- per D-12 the delete button does a server-side dependency check on click. So the flow is:
1. User clicks Delete button -> calls `handleDelete(role)` directly
2. handleDelete calls the DELETE API endpoint
3. If 409: show DeleteBlockedModal with dependency info
4. If 200: show success toast (this means it was deleted because the server found no deps and deleted it)

Actually re-reading D-12 and D-15 more carefully: the server checks dependencies and either blocks (409) or succeeds (200). But D-15 says "If no dependencies exist, a standard confirmation dialog asks 'Delete [role/permission name]? This action cannot be undone.' with Cancel/Delete buttons." This means the confirmation dialog should appear BEFORE the actual delete when there are no dependencies. So the flow should be:

1. User clicks Delete button -> calls the DELETE API endpoint to CHECK dependencies
2. If 409: show DeleteBlockedModal
3. If 200 (no deps, but item not yet deleted): show DeleteConfirmModal, on confirm -> call delete again

Wait, that requires two endpoints or two behaviors. Looking at the RESEARCH.md recommendation in Open Question 1: "Use a single DELETE endpoint that returns 409 Conflict with dependency data when blocked, and performs the deletion when clear." This means 200 = already deleted, no confirmation step.

But D-15 explicitly says show a confirmation before deleting. We need to reconcile: the UX requires a confirmation step for items without dependencies. So the correct flow is:

1. User clicks Delete -> first make a lightweight check call, OR use the DELETE endpoint but need a way to ask for check-only.

Simplest approach: Add a query param `?check=true` to the DELETE endpoint that only checks dependencies without deleting. Or create a separate check endpoint. Alternatively, the cleanest approach per the existing API patterns: the client first calls DELETE. If 409, show blocked modal. If 200, the item is already deleted. This skips the D-15 confirmation, which contradicts the requirement.

**Resolution:** Implement a two-phase delete on the client side:
1. Click Delete -> call `fetch(\`/api/db/admin/roles/\${role.roleID}?check=true\`, { method: 'DELETE', ... })`
2. The controller checks: if `req.query.check === 'true'`, perform dependency check only and return `{ canDelete: true }` or 409 with deps -- do NOT actually delete.
3. If `canDelete: true`: show DeleteConfirmModal. On confirm: call DELETE again WITHOUT `?check=true` to perform actual deletion.
4. If 409: show DeleteBlockedModal.
