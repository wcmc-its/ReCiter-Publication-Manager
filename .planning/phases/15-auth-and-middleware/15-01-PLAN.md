<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> JWT cookie | Client receives httpOnly cookie; cannot read or modify JWT contents |
| JWT callback -> Database | Server-side callback queries MySQL for permission resolution |
| Database -> JWT payload | Query results are serialized into the JWT token as JSON strings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | Tampering | JWT permissions field | mitigate | JWT is signed by next-auth using NEXTAUTH_SECRET. The permissions field is set server-side in the JWT callback and serialized into the signed token. Clients cannot modify it because the cookie is httpOnly and the JWT signature prevents tampering. |
| T-15-02 | Injection | findUserPermissionsEnriched SQL | mitigate | All user-supplied values (email, personIdentifier) are passed as Sequelize named replacements (:paramName), never string-concatenated into SQL. This is identical to the existing findUserPermissions pattern that has been in production. |
| T-15-03 | Elevation of Privilege | Empty permissions fallback | accept | If findUserPermissionsEnriched fails (try/catch), the user gets empty permissions/permissionResources arrays. The middleware (Plan 02) applies baseline fallback (canSearch + canReport only) — no access to admin routes. This matches current behavior for users with no roles. |
| T-15-04 | Information Disclosure | Permission data in JWT | accept | Permission keys (canCurate, canSearch, etc.) and resource metadata (nav labels, routes, icons) are not sensitive — they describe the app's menu structure. Even if a user could decode the JWT, this data provides no advantage since the server enforces access. |
| T-15-05 | Denial of Service | Double DB query on login | accept | findUserPermissionsEnriched adds two SQL queries at login time. Login is infrequent (once per session, max every 2 hours per JWT maxAge). The queries use indexed JOINs on small tables (7 permissions, 18 mappings). Impact is negligible (~5ms added to login). |
</threat_model>
