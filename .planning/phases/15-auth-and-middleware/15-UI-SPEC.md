> Behavioral UI contract for Phase 15: Auth and Middleware.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Phase Surface Area: BEHAVIORAL (No New Visual Elements)

Phase 15 rewires the auth pipeline and middleware to use database-driven permissions instead of hardcoded role-label checks. **No new pages, no new components, no visual changes, no layout changes are in scope.** The user-visible behavior -- what they see during login, redirect, and access denial -- must remain pixel-identical.

Evidence:
- ROADMAP.md Phase 15 goal: "all without changing any visible behavior"
- RESEARCH.md summary: "The goal is zero visible behavior change"
- ROADMAP.md: Phase 15 has no `UI hint` marker (contrast with Phases 16 and 17 which do)
- Requirements AUTH-01 through MW-05 are auth pipeline, JWT, and middleware concerns
- No new pages, routes, or components are created

This UI-SPEC defines the **behavioral contracts** that Phase 15 must preserve. These are the user-visible behaviors that the executor must verify after implementation. Downstream phases (16-18) reference this document for the session data shape they will consume.

## Behavioral Contract: Login Flow

Phase 15 must preserve identical user-visible login behavior for all authentication paths. The internal data enrichment (adding `permissions` and `permissionResources` to the JWT) is invisible to the user.

### Direct Login (credentials)

| Step | Current Behavior | Phase 15 Behavior | Visible Change |
|------|-----------------|-------------------|----------------|
| 1. User enters CWID + password | Login form at `/login` | Unchanged | None |
| 2. Successful auth | Toast: "Successful Login" (green, top-right, 2s auto-close) | Unchanged | None |
| 3. Session established | JWT contains `userRoles` | JWT contains `userRoles` + `permissions` + `permissionResources` | None (JWT is httpOnly cookie) |
| 4. Post-login redirect | Role-based redirect (see Redirect Matrix below) | Permission-based redirect producing identical destinations | None |
| 5. Failed auth | Toast: "Invalid credentials" (red, top-right, 2s auto-close) | Unchanged | None |

### SAML Login

| Step | Current Behavior | Phase 15 Behavior | Visible Change |
|------|-----------------|-------------------|----------------|
| 1. Redirect to IdP | `/api/saml/assert?callbackUrl=/search` | Unchanged | None |
| 2. IdP authenticates | External SAML flow | Unchanged | None |
| 3. Session established | JWT contains `userRoles` | JWT contains `userRoles` + `permissions` + `permissionResources` | None |
| 4. Post-login redirect | Role-based redirect (see Redirect Matrix below) | Permission-based redirect producing identical destinations | None |

### Critical Invariant

The Login component (`src/components/elements/Login/Login.js`) is NOT modified in this phase. It continues to call `getSession()` and read `session.data.userRoles` for post-login redirect logic. Phase 16 will update it to use `session.data.permissions`.

### Index Page Redirects (`/` -> getServerSideProps)

| Role Combination | Permissions Resolved | Landing Page | Source |
|-----------------|---------------------|--------------|--------|
| Unauthenticated (no session) | n/a | `/login` (non-SAML) or `/api/saml/assert` (SAML) | src/pages/index.js lines 59-64, 101-106 |
| Authenticated, status=0 (inactive) | n/a | `/noaccess` | src/pages/index.js lines 27-31, 69-73 |
| Curator_Self only | `['canCurate']` | `/curate/{personIdentifier}` | src/pages/index.js lines 34-39, 76-80 |
| Curator_Self + Reporter_All | `['canCurate', 'canSearch', 'canReport']` | `/curate/{personIdentifier}` | Self-role takes priority per getLandingPageFromPermissions |

| No permissions (baseline) | `/report` | canReport: YES (baseline) | Allow through | MW-03 fallback |

| No session cookie | Any matched route | n/a | `/login` | middleware.ts cookie check |

### Critical Self-Only Detection Rule

The middleware must use **role labels** (from `userRoles` JWT field), not permission keys, to detect self-only curators. Both Curator_Self and Curator_Scoped resolve to `canCurate`, but only Curator_Self should be restricted to their own curate page. The RESEARCH.md Pitfall 1 documents this explicitly. The detection logic checks:
1. User has a role with `roleLabel === 'Curator_Self'`
2. User does NOT have any role in `['Superuser', 'Curator_All', 'Curator_Scoped', 'Curator_Department', 'Curator_Department_Delegate']`
3. If both conditions true, redirect `/curate/{otherId}` to `/curate/{ownId}`

## Behavioral Contract: Access Denial

### No Access Page (`/noaccess`)

The NoAccess component displays one of two messages based on session state. Phase 15 must not alter this behavior.

| Condition | Copy Displayed | Source |
|-----------|---------------|--------|
| Authenticated, has roles, status=0 | "Sorry, you don't have access to the system. Please, contact your administrator." | NoAccess.tsx line 22 |
| Authenticated, no roles assigned | "You have successfully authenticated, but you don't have any roles assigned. Please contact a system administrator" | NoAccess.tsx line 21 |

**Styling contract (must remain unchanged):**
- Container: centered horizontally, full viewport height (`100vh`)
- Text: red color (`color: red`), centered, 20px font size, 25px top margin

### Error Redirect

The middleware redirects to `/error` when no roles are found in the JWT (middleware.ts line 122). There is no `/error` page in the codebase -- this produces a Next.js 404. Phase 15 must preserve this behavior (the middleware continues to use `userRoles` for the no-roles check until Phase 18 cleanup).

### Forbidden Response

Requests containing `.git` in the path receive a 403 status with no body. This behavior is unchanged.

## Behavioral Contract: Session Data Shape

Phase 15 adds two new fields to the session. These are consumed by Phase 16 components. Phase 15 itself does not render these fields, but the data contract must be documented for downstream consumers.

### JWT Token Fields (after Phase 15)

| Field | Type (in JWT) | Type (parsed) | Example | Added By |
|-------|--------------|---------------|---------|----------|
| `userRoles` | JSON string | `Array<{personIdentifier, roleLabel, roleID}>` | `'[{"personIdentifier":"abc123","roleLabel":"Superuser","roleID":1}]'` | Existing (kept for backward compat) |
| `permissions` | JSON string | `string[]` | `'["canCurate","canSearch","canReport","canManageUsers","canConfigure","canManageNotifications","canManageProfile"]'` | NEW in Phase 15 |
| `permissionResources` | JSON string | `Array<{resourceType, resourceKey, displayOrder, icon, label, route, permissionKey}>` | `'[{"resourceType":"nav","resourceKey":"search","displayOrder":1,"icon":"Search","label":"Search","route":"/search","permissionKey":"canSearch"}]'` | NEW in Phase 15 |

### Client-Side Access Pattern

```
session.data.permissions       -> JSON string, parse with JSON.parse() or getPermissionsFromRaw()
session.data.permissionResources -> JSON string, parse with JSON.parse()
session.data.userRoles         -> JSON string (UNCHANGED, kept for Phase 16 transition)
```

### Permission Keys (complete set from seed data)

| Permission Key | Granted To Roles | Route Access |
|---------------|-----------------|--------------|
| `canCurate` | Superuser, Curator_All, Curator_Self | `/curate/*` |
| `canSearch` | Superuser, Curator_All, Reporter_All | `/search` |
| `canReport` | Superuser, Reporter_All | `/report` |
| `canManageUsers` | Superuser | `/manageusers` |
| `canConfigure` | Superuser | `/configuration` |
| `canManageNotifications` | Superuser, Curator_All, Curator_Self | `/notifications/*` |
| `canManageProfile` | Superuser, Curator_All, Curator_Self, Reporter_All | `/manageprofile/*` |

### Route-to-Permission Map (MW-02)

This is the single lookup object that replaces the 130-line if/else chain in middleware.ts:

```typescript
const ROUTE_PERMISSIONS: Record<string, string> = {
  '/manageusers': 'canManageUsers',
  '/configuration': 'canConfigure',
  '/curate': 'canCurate',
  '/report': 'canReport',
  '/search': 'canSearch',
  '/notifications': 'canManageNotifications',
  '/manageprofile': 'canManageProfile',
}
```

## Interaction Patterns

No new interactions are introduced. The following existing interaction patterns must work identically:

| Interaction | Behavior | Verification Method |
|-------------|----------|-------------------|
| Submit login form | POST credentials, toast feedback, redirect to landing page | Manual test or Playwright: fill form, submit, assert URL |
| SAML redirect | Browser redirects to IdP, returns to app, lands on correct page | Manual test: initiate SAML flow, verify landing |
| Navigate to restricted route | Middleware redirects to user's landing page | Playwright: navigate to `/manageusers` as Curator_Self, assert redirect to `/curate/{id}` |
| Navigate to own curate page | Page loads normally | Playwright: navigate to `/curate/{ownId}` as Curator_Self, assert page renders |
| Navigate to other's curate page (self-only) | Middleware redirects to own curate page | Playwright: navigate to `/curate/other` as Curator_Self, assert redirect |
| Access with no session | Middleware redirects to `/login` | Playwright: clear cookies, navigate to `/search`, assert redirect to `/login` |
