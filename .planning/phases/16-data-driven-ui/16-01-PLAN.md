**Per Claude's Discretion: Create src/hooks/usePermissions.ts:**

Create a `src/hooks/` directory if it does not exist. The hook wraps session access, permission parsing, and permissionResources parsing in memoized values:

```typescript
import { useMemo } from 'react'
import { useSession } from 'next-auth/client'
import { getPermissionsFromRaw, hasPermission } from '../utils/permissionUtils'

export interface PermissionResource {
  resourceType: string
  resourceKey: string
  displayOrder: number
  icon: string
  label: string
  route: string
  permissionKey?: string
}

export function usePermissions() {
  const [session, loading] = useSession()

  const permissions = useMemo(
    () => getPermissionsFromRaw(session?.data?.permissions),
    [session?.data?.permissions]
  )

  const permissionResources = useMemo(() => {
    try {
      const raw = session?.data?.permissionResources
      return typeof raw === 'string' ? JSON.parse(raw) : (raw || [])
    } catch {
      return []
    }
  }, [session?.data?.permissionResources])

  const check = (key: string) => hasPermission(permissions, key)

  return { permissions, permissionResources, hasPermission: check, session, loading }
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| JWT -> Client UI | permissionResources and permissions are signed JWT claims rendered in browser |
| Client UI rendering | Nav visibility is UX convenience, not security enforcement |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-16-01 | Elevation of Privilege | SideNavbar renders nav items based on client-side permission check | accept | Client-side UI gating is supplementary UX only. Server-side enforcement exists in Edge middleware (Phase 15, ROUTE_PERMISSIONS map) and API route Authorization headers. A user who bypasses nav hiding still hits server-side 403. |
| T-16-02 | Tampering | permissionResources in JWT could be modified | mitigate | JWT is signed by next-auth with NEXT_PUBLIC_RECITER_TOKEN_SECRET (HMAC). Client cannot modify claims without invalidating the signature. Existing next-auth verification in middleware.ts validates the token on every request. |
| T-16-03 | Information Disclosure | Empty/malformed permissionResources could render unexpected UI | mitigate | usePermissions() hook catches JSON.parse errors and returns empty array. getPermissionsFromRaw() returns empty array on any error. hasPermission() returns false for empty arrays. Default-deny: no permissions = no nav items visible. |
</threat_model>
