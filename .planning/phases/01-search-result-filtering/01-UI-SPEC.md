## Spacing Scale

Declared values (must be multiples of 4). Derived from existing codebase patterns in `globals.css`, `AppLayout.module.css`, and component CSS modules:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline gaps (e.g., badge margin from text) |
| sm | 8px | Compact element spacing, skeleton line gaps |
| md | 16px | Default element spacing, skeleton block gaps |
| lg | 20px | Section padding (existing: main content uses 20px 40px) |
| xl | 32px | Layout gaps, page horizontal padding matches existing 40px where contextual |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing (not used in this phase) |

Exceptions:
- Main content area uses `padding: 20px 40px` (AppLayout.module.css) -- 40px horizontal is the established pattern, not a strict multiple of 4 but a multiple of 8. Preserve as-is.
- Existing filter sections use `padding: 15px` and `padding: 10px 30px` -- preserve these existing values, do not refactor spacing in existing components.
- Skeleton components use the same padding/margin as the content they replace (matched to target component dimensions).

## Skeleton Component Specification

### Animation Style

**Shimmer** (CSS-only, no library). A left-to-right shine effect using CSS `@keyframes` and `linear-gradient`.

```css
/* Skeleton base class -- all skeleton components use this */
.skeleton {
  background-color: #e0e0e0;
  background-image: linear-gradient(90deg, #e0e0e0 25%, #f0f0f0 50%, #e0e0e0 75%);
  background-size: 200% 100%;
  animation: shimmer 1.5s ease-in-out infinite;
  border-radius: 4px;
}

@keyframes shimmer {
  0% { background-position: 200% 0; }
  100% { background-position: -200% 0; }
}
```

Source: CONTEXT.md grants Claude's discretion on "Skeleton animation style (pulse vs shimmer)". Shimmer chosen because it provides clearer visual feedback that content is loading, versus pulse which can be mistaken for a static placeholder.

### SkeletonTable

Renders a placeholder matching the search results table or manage users table layout:
- 1 header row: single rectangle, full width, height 40px, background `#2E414F` at 30% opacity
- 5 body rows: each row contains 3-4 rectangles (matching column count), height 20px, spaced 12px apart vertically
- Table container: matches `.table` border styling (1px solid border)
- Total height: approximately matches 5 rows of real data

### SkeletonCard

Renders a placeholder matching a publication card:
- Title line: rectangle, 60% width, height 16px
- 2 metadata lines: rectangles, 40% and 30% width, height 12px, 8px gap
- Button area: 2 small rectangles (60px wide, 28px tall), right-aligned, 8px gap
- Repeat 3 times vertically with 16px gap between cards

### SkeletonProfile

Renders a placeholder matching the curate page identity header:
- Avatar circle: 80px diameter circle
- Name line: rectangle next to avatar, 200px wide, height 20px
- Title line: rectangle, 150px wide, height 14px, 8px below name
- Metadata row: 3 small rectangles (100px, 80px, 120px), height 12px, 8px below title
- Keywords section: 5 small pill shapes (60-100px wide, 24px tall), 8px gap, 16px below metadata

### SkeletonForm

Renders a placeholder matching form/settings layouts:
- 4 rows, each with: label rectangle (80px wide, 14px tall) and input rectangle (full width, 36px tall)
- 16px gap between rows
- 1 button rectangle at bottom right (100px wide, 36px tall)

### Skeleton Sizing

CONTEXT.md grants Claude's discretion on "Whether to add size prop to skeleton components." Decision: No size prop. Each skeleton component is purpose-built to match its target content dimensions. Simpler API, less ambiguity for implementers.

## Interaction Contracts

### Search Page Dropdown (UIBUG-01)

**Before:** Dropdown contains "Curate publications", "View Profile", "View Report" (varies by role).
**After:** Dropdown contains "View Profile", "View Report" only. "Curate publications" removed for ALL users including Superuser.

- Curators click person name to navigate to `/curate/:id` (existing behavior, preserved).
- Non-curators click person name to go to Create Reports with that person selected (existing behavior, preserved).
- Dropdown remains a `SplitDropdown` react-bootstrap component.
- No animation on removal. Static change.

### Profile Modal (UIBUG-02)

**Success state:** Modal opens with biographical and bibliometric data. No change to existing layout.

**Error state:** Instead of generic "Something went wrong", show: "Unable to load profile data. The person record may be incomplete or temporarily unavailable. Try again or contact your administrator." displayed in the modal body area, 14px, color `#dc3545`, centered.

**PII stripping for non-curators:**
- Hide: email addresses, relationship data
- Show: name, title, department, publication metrics, keywords, article scores
- Implementation: conditional rendering in Profile.tsx based on capability check (`canCurate`)

### Loading State Transitions (UIBUG-03)

**Before:** All loading states show `<Loader />` (centered react-bootstrap Spinner, variant="danger", 5rem).
**After:** Each page shows an appropriate skeleton component that matches the shape of the content being loaded.

Transition: Skeleton appears immediately on mount/data fetch. When data arrives, skeleton is replaced by real content with no animation (instant swap). No fade-in, no crossfade. Matches existing `<Loader />` replacement pattern (conditional render).

Pages and their skeleton assignments:

| Page | Current Loading | New Loading | Skeleton Component |
|------|----------------|-------------|-------------------|
| Search (`/search`) | `<Loader />` | Skeleton table | `<SkeletonTable />` |
| Curate (`/curate/:id`) | `<Loader />` | Skeleton profile + skeleton cards | `<SkeletonProfile />` + `<SkeletonCard />` |
| Manage Users (`/manageusers`) | `<Loader />` | Skeleton table | `<SkeletonTable />` |
| Create Reports (`/report`) | `<Loader />` | Preserve existing pattern (already has skeleton-like loading) | No change |
| Configuration (`/configuration`) | `<Loader />` | Skeleton form | `<SkeletonForm />` |
| Notifications (`/notifications`) | `<Loader />` | Skeleton form | `<SkeletonForm />` |

### Auth Flow (AUTH-01 through AUTH-04)

No visual UI changes. All auth changes are backend logic (middleware, JWT callbacks, routing). The only user-visible effect is: users who were incorrectly redirected to `/noaccess` will now reach their correct landing page.
